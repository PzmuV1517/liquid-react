# Thanks for your interest in contributing

Hello! We appreciate your interest in contributing to the liquidReact library. We welcome contributions of all kinds, including but not limited to bug fixes, new features, documentation improvements, code refactoring, suggestions, and more, especially since we are a group of teenagers who also have to focus on their personal lives. To ensure a smooth collaboration process, we have outlined some guidelines and best practices for contributing to this project. Please take a moment to review the following information before submitting your contribution. Aside from the guidelines below, we also encourage you to check out our [Code of Conduct](.github/CODE_OF_CONDUCT.md) to ensure a positive and friendly environment for all contributors.

## First off, how do I contribute?

Contributing to liquidReact is just like any other open source project. There are 2 main methods you can contribute:

**Pull Requests:** After you figure out how and what you want to add/change/remove/etc, you can fork the repository, make your changes in a *new branch* (this is mandatory), and then submit a pull request to *staging* (not main). It is best to submit your pull request as a draft after the first commit, and fill out the pull request template with the necessary information. We will review your pull request and provide feedback or merge it if it meets our standards.

**Issues:** Another way to contribute is to simply submit an issue in the repository's issue tracker. If you find a bug or have a feature request, please submit it there so that we can track and address it. Make sure it follows the issue template and provides enough information for us to understand and reproduce the problem or implement the feature.

## Ok, but what are the guidelines?

We have a few standards that we ask all contributors to follow when submitting a pull request. These standards are in place to ensure that our codebase remains clear, consistent and maintainable for future helpers. It's also to make sure that no arguments arise (for example from a duplicate pull request or issue).

**1. Follow the Code of Conduct:** We expect all contributors to adhere to our [Code of Conduct](.github/CODE_OF_CONDUCT.md) to maintain a respectful and inclusive environment for everyone.

**2. Make sure you know what you're doing:** All code, documentation, and other contributions you make should be well thought out and intentional. If you are unsure about how to implement a feature or fix a bug, please ask for help in the discussions or open an issue to get feedback before submitting a pull request.

**2.1. No AI Generated Content:** We prohibit AI-generated content in contributions to this project (with some exceptions for trusted contributors). This is because AI-generated content is often inaccurate, lacks context, clouds code quality (which can lead to confusion and bugs) and can sometimes infringe on intellectual property rights (we cannot verify the source of the model's training data, which could lead to unwanted legal issues). We encourage you to write your own original content to ensure the quality and integrity of our project.

**3. Make sure your PR passes CI/CD checks:** We have automated checks in place to ensure that all contributions meet our quality standards. Please make sure your pull request passes all checks before submitting it. This includes linting, testing, and other checks that we have set up.

**4. Follow the pull request template:** When submitting a pull request, please fill out the pull request template with all the necessary information. This helps us understand the purpose and scope of your contribution and makes it easier for us to review and merge it.

**5. Be responsive to feedback:** We may provide feedback on your pull request, and we ask that you be responsive and open to making changes based on that feedback. This helps us maintain a high-quality codebase and ensures that your contribution is in line with our standards.

**6. Be patient:** We receive many contributions, and it may take some time for us to review and merge your pull request. Please be patient and understand that we are doing our best to review all contributions in a timely manner.

**7. Considering:** Considering the fact that this library is a strictly component-based library, we ask that, unless it's a documentation or bug fix contribution, all contributions should be in the form of a new component or an improvement to an existing component. This is not a hard rule, but it is a general guideline to help maintain the focus and structure of our library.

**7.1. NOTE:** As the library is focused specifically on iOS Liquid-Glass components, which can be difficult to implement and often require a lot of knowledge on how the infrastructure of the library works, we ask that all contributions related to new components (this doesn't include improvements, bug fixes, documentation, etc) be made by trusted contributors and/or maintainers. This is not a hard rule, but if you're unexperienced with the library, it's easier to let someone with more expertise handle the implementation of new components so no confusion arises from who does what and no duplicate pull requests come up.

## Style guidelines

When contributing to Liquid-React, please follow our style guidelines to ensure that your code is consistent with the rest of the library. This includes using clear and descriptive variable names, writing comments where necessary, and following the existing code structure.

- Use camelCase for variable and function names.
- Use PascalCase for component names.
- Write clear and concise comments to explain the purpose of your code.
- All commits should be prefixed with the type of change in this format:
```txt
[fix/feat/refactor/docs] Short description of the change
```
- Use the present tense in commit messages (e.g., "Add new feature" instead of "Added new feature").
- Keep commit messages concise and to the point, ideally under 50 characters for the subject line and 72 characters for the body.
- Use present tense in commit messages (e.g., "Add new feature" instead of "Added new feature").
- Use eslint and prettier to ensure that your code is properly formatted and follows our style guidelines.
